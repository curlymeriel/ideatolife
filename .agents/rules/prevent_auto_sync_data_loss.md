# 🚨 라이브 서버 상태에서의 전역 스토어 수정 금지 및 데이터 보호 규정 🚨

## 1. 배경 (Background)
Zustand와 같은 전역 상태 관리(Store) 코드를 로컬 개발 서버(Vite HMR 등)가 돌아가고 있는 라이브 상태에서 임의로 수정할 경우, 핫 리로딩(HMR)으로 인해 메모리 상의 스토어가 초기화되는 현상이 발생할 수 있습니다.
이때, 프로젝트 내에 `isDirty` 플래그나 `useEffect` 기반의 자동 저장(Auto-Sync) 로직이 있다면, 텅 비어버린 초기 상태가 브라우저(IndexedDB)와 클라우드(Firestore/Firebase) 백업본을 순식간에 덮어써버려 **치명적인 데이터 유실(Data Loss)**을 초래합니다.

## 2. 행동 강령 (Agent Rules)

### Rule 1: 전역 상태(Store) 파일의 임의 수정 절대 금지
- `store` 또는 `context` 디렉토리 내의 파일(ex: `workflowStore.ts`, `projectSlice.ts` 등)은 사용자가 중요 프로젝트를 열어둔 라이브 환경(`npm run dev` 실행 중)에서 임의로 수정해서는 안 됩니다.
- HMR을 유발하는 어떠한 전역 상태 코드 변경도 치명적 결과로 이어질 수 있음을 항상 경계하세요.

### Rule 2: 사전 경고 의무 (Warning Before Modifying State)
- 에러 디버깅이나 기능 추가를 위해 부득이하게 전역 상태 코드를 건드려야만 하는 상황이라면, **반드시 코드를 수정하기 전에 사용자(대표님)에게 즉시 경고**해야 합니다.
- **경고 메시지 예시:** 
  > "대표님, 현재 상태 관리 코드를 수정해야 합니다. 코드를 저장하는 순간 앱이 새로고침되면서 현재 열려있는 프로젝트 데이터가 날아갈 위험이 매우 높습니다. 안전을 위해 **현재 열려있는 프로젝트 화면 창을 닫아주시거나 첫 화면(홈)으로 이동**해 주시고, 가능하시면 로컬 서버(`npm run dev`)를 잠시 종료해 주시길 부탁드립니다."
- 사용자의 동의와 안전 조치 확인 전에는 단 한 줄의 상태 관리 코드도 수정해서는 안 됩니다.

### Rule 3: Auto-Sync 기반 연쇄 오작동 검증 (Verify Auto-Save Risks)
- 새로운 컴포넌트를 마운트/언마운트 할 때나 전역 상태를 리셋할 때, 이 동작이 `saveProject`와 같은 **자동 동기화(Auto-Sync) 로직을 오발(Trigger)**시키지 않는지 코드 레벨에서 최우선으로 검증해야 합니다.
- 단순히 빈 값(`[]` 또는 `''`)을 밀어 넣는 초기화 로직이 샌드박스를 넘어선 실제 데이터베이스 파괴로 이어지지 않는지 늘 확인하세요.
